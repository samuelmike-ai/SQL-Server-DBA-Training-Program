# SQL Server DBA Portfolio Guidelines

## Overview

A professional portfolio is essential for SQL Server Database Administrator career advancement, serving as tangible evidence of technical expertise, problem-solving abilities, and professional growth. This comprehensive guide provides structured approaches to building, organizing, and presenting a DBA portfolio that demonstrates competency and value to potential employers and clients.

## Portfolio Philosophy

Our portfolio development approach emphasizes:
- **Professional Credibility:** Demonstrating real-world expertise and results
- **Measurable Impact:** Quantifying contributions and business value
- **Technical Depth:** Showcasing advanced skills and continuous learning
- **Career Progression:** Illustrating growth and development trajectory
- **Future-Ready:** Highlighting adaptability and emerging technology adoption

### Monitoring Solution Documentation

#### Dashboard Configuration Examples

**Performance Monitoring Dashboard:**
```sql
-- =============================================
-- Performance Monitoring Query Set
-- =============================================
-- Purpose: Comprehensive performance monitoring queries
-- Usage: Run queries regularly and log results for trending
-- =============================================

-- 1. Current Database Performance
SELECT 
    database_name = DB_NAME(database_id),
    file_type = CASE WHEN type_desc = 'ROWS' THEN 'Data File' ELSE 'Log File' END,
    file_size_mb = CAST(CAST(size AS BIGINT) * 8 / 1024 AS DECIMAL(18,2)),
    space_used_mb = CAST(CAST(FILEPROPERTY(name, 'SpaceUsed') AS BIGINT) * 8 / 1024 AS DECIMAL(18,2)),
    free_space_mb = CAST((CAST(size AS BIGINT) - CAST(FILEPROPERTY(name, 'SpaceUsed') AS BIGINT)) * 8 / 1024 AS DECIMAL(18,2)),
    percent_used = CAST((CAST(FILEPROPERTY(name, 'SpaceUsed') AS BIGINT) * 100.0 / CAST(size AS BIGINT)) AS DECIMAL(5,2))
FROM sys.database_files
WHERE type IN (0,1)
ORDER BY database_name, type_desc;
```

**Blocking Detection Query:**
```sql
-- =============================================
-- Blocking Detection and Analysis
-- =============================================
-- Purpose: Identify and analyze blocking scenarios
-- Usage: Execute when performance issues are reported
-- =============================================

SELECT 
    blocked_session.session_id AS blocked_session_id,
    blocked_session.login_name AS blocked_user,
    blocked_session.host_name AS blocked_host,
    blocked_session.program_name AS blocked_program,
    blocking_session.session_id AS blocking_session_id,
    blocking_session.login_name AS blocking_user,
    blocking_session.host_name AS blocking_host,
    blocking_session.program_name AS blocking_program,
    waitstats.wait_type,
    waitstats.wait_duration_ms,
    sql_text.text AS blocking_query_text,
    blocked_sql.text AS blocked_query_text
FROM sys.dm_exec_requests blocked_request
INNER JOIN sys.dm_exec_sessions blocked_session ON blocked_request.session_id = blocked_session.session_id
INNER JOIN sys.dm_exec_sessions blocking_session ON blocked_request.blocking_session_id = blocking_session.session_id
INNER JOIN sys.dm_os_wait_stats waitstats ON blocking_session.wait_type = waitstats.wait_type
OUTER APPLY sys.dm_exec_sql_text(blocking_session.sql_handle) sql_text
OUTER APPLY sys.dm_exec_sql_text(blocked_session.sql_handle) blocked_sql
WHERE blocked_request.blocking_session_id != 0
ORDER BY waitstats.wait_duration_ms DESC;
```

### Knowledge Base Development

#### Standard Operating Procedures (SOPs)

**Database Backup Procedure SOP:**
```markdown
# Standard Operating Procedure: Database Backup

## Purpose
This SOP ensures consistent, reliable backup procedures for all production databases, maintaining business continuity and data protection standards.

## Scope
Applies to all SQL Server production databases and backup operations.

## Responsibilities
- Database Administrator: Execute backup procedures
- Backup Administrator: Monitor backup completion and success
- Operations Manager: Review backup reports and address failures

## Procedure
1. **Pre-Backup Verification**
   - Verify database is online and accessible
   - Check available disk space (minimum 150% of database size)
   - Confirm backup location is accessible

2. **Backup Execution**
   - Execute full backup using standardized script
   - Monitor backup progress and completion
   - Verify backup file creation and size

3. **Post-Backup Validation**
   - Verify backup file integrity using RESTORE VERIFYONLY
   - Confirm backup completion in backup history
   - Document any issues or anomalies

4. **Reporting**
   - Generate backup success/failure report
   - Distribute report to operations team
   - Address any failed backups within 2 hours

## Quality Control
- Weekly backup validation testing
- Monthly backup recovery testing
- Quarterly documentation review and update

## Related Documents
- Disaster Recovery Procedures
- Backup Retention Policy
- Emergency Contact List
```

#### Troubleshooting Guides

**Performance Issue Troubleshooting Guide:**
```markdown
# Performance Issue Troubleshooting Guide

## Quick Assessment Checklist
- [ ] Check current wait statistics and resource utilization
- [ ] Identify active blocking scenarios
- [ ] Review recent changes to database or application
- [ ] Analyze query performance baselines
- [ ] Verify system resource availability (CPU, memory, I/O)

## Common Performance Issues and Solutions

### Slow Query Performance
**Symptoms:** Increased response times, timeout errors
**Immediate Actions:**
1. Check execution plans for problematic queries
2. Identify missing or inefficient indexes
3. Review recent statistics updates
4. Check for blocking and waiting processes

**Resolution Steps:**
1. Optimize query structure and syntax
2. Create or modify indexes based on query patterns
3. Update statistics and rebuild fragmented indexes
4. Consider query hints or plan guides if needed

### Database Connectivity Issues
**Symptoms:** Connection timeouts, authentication failures
**Immediate Actions:**
1. Verify SQL Server service status
2. Check network connectivity and firewall settings
3. Review SQL Server error logs
4. Test connection from multiple clients

**Resolution Steps:**
1. Restart SQL Server services if necessary
2. Review and update connection string settings
3. Check authentication and authorization configuration
4. Implement connection pooling and timeout optimization

## Escalation Procedures
1. **Level 1:** DBA team investigation and resolution (30 minutes)
2. **Level 2:** Senior DBA and management notification (1 hour)
3. **Level 3:** Vendor support and executive escalation (2 hours)
```
